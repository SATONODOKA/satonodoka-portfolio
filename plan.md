# ポートフォリオサイト開発計画

## 1. プロジェクトの目的と評価軸

### 目的
**「専門エンジニアではない人物が、短期間で"使えるプロダクト"を自力で作り、改善し、公開できるフットワークと思考力を持っている」**という印象を、転職活動における一次接触（流し見）で伝えること。

### 評価されたいポイント
- 完成度よりも **構造理解と意思決定の筋**
- 技術力そのものではなく **問題設定 → 仮説 → 最小実装 → 改善** の再現性
- 作るスピード感・公開まで持っていく行動力
- 「この人にツールを持たせたら、業務改善やプロダクト検討を前に進めそう」という期待

### 非目標
- 技術スタック詳細説明・ライブラリ羅列・コード自慢
- 個人ブログ
- 派手なアニメーション
- デザインで驚かせること

## 2. 技術スタック提案

### フロントエンド
- **Next.js 14+ (App Router)**: 静的生成と動的ルーティング、SEO最適化
- **TypeScript**: 型安全性と保守性
- **Tailwind CSS**: ユーティリティファースト、ミニマルデザインの実現
- **Framer Motion**: 軽量なアニメーション（フェードイン、スライド）

### スタイリング
- **CSS Modules / Tailwind**: コンポーネント単位のスタイリング
- **12カラムグリッドシステム**: Tailwindのグリッド機能を活用

### 画像・アセット
- **Next.js Image**: 最適化された画像配信
- **SVG**: ロゴ・アイコン

### デプロイ
- **Vercel**: Next.jsとの統合、自動デプロイ

### 開発環境
- **ESLint + Prettier**: コード品質
- **Git**: バージョン管理

## 3. 全体構成（サイトマップ）

以下の2ページ構成：

1. **Top / Home** (`/`) - すべてのセクションを含む単一ページ
   - Hero セクション
   - Featured Projects セクション（アンカー: `#work`）
   - How I Work セクション
   - Artifacts / Process Proof セクション
   - About セクション（アンカー: `#about`）
2. **Case Study** (`/work/[slug]`) - 個別詳細ページ（テンプレート1種）

※ プロジェクト詳細以外はページ遷移不要。すべてTopページ内のセクションとして実装。
※ 「作品を見る」ボタンはスクロールで `#work` セクションまで移動。
※ ナビゲーションもアンカーリンクで各セクションに移動。

## 4. 共通デザイン方針

### トーン & コンセプト
- ミニマル、情報設計重視
- 「SaaS企業のプロダクト紹介ページ」に近い空気感
- 装飾で魅せない。**余白・整列・階層**で魅せる
- 個人ポートフォリオ特有の"自己主張の強さ"は避ける

### レイアウト
- デスクトップ：**12カラムグリッド**
- 最大横幅は制限（読みやすさ優先、ワイドすぎない）
- セクション間マージンは大きめ（視線のリセットを重視）

### タイポグラフィ
- 見出し（H1〜H3）と本文の階層を明確に
- H1：太め・大きめ・1〜2行
- 本文：行間広め、可読性最優先
- 数字・期間・回数は「スキャンしやすいUI要素」として扱う

### カラー
- ベースはモノトーン（白〜薄グレー）
- アクセントカラーは1色のみ（CTA・リンク・強調用）
- グラデーションや派手な配色は禁止

### コンポーネント指針

#### Card UI
- 角丸は中程度
- シャドウは極薄（Elevation低）
- 情報は「タイトル → 補足 → メタ情報」の縦構造

#### Button
- Primary：塗り
- Secondary：アウトライン
- サイズ・余白は明確に差をつける

#### Animation
- フェードイン、軽いスライドのみ
- スクロールトリガー
- 派手な動きは禁止

## 5. Top / Home ページ設計

### 5-1. Hero セクション（ファーストビュー）

**2カラム構成（PC）／縦積み（SP）**

左側：
- 小さな肩書きテキスト（カテゴリ的なラベル）
- 大きなH1コピー（価値主張）
- 補足説明文（2行程度、抽象すぎない）
- CTAボタン2つ
  - Primary：作品を見る
  - Secondary：話す／相談する

右側：
- 自作プロダクトの画面キャプチャを **3枚コラージュ**
- ノートPC / スマホのモックフレームを使用
- 各画像に小さなタグ（例：`Shipped in 7 days`、`Iteration x5`）

※ この時点で「作って終わりではない人」という印象を作る

### 5-2. Featured Projects セクション

- 代表プロジェクト3件
- カードUIで横並び（SPは縦）

各カードに含める情報構造：
- プロジェクト名
- 一言での目的 or 効果
- 制作期間（例：5 days）
- タグ（抽象度高め、技術名は使わない）
- サムネイル（UI画面 or 構成図）

※ 「完成度」ではなく「意図とスピード」が伝わることを重視

### 5-3. How I Work セクション

**3ステップ構成（横並び）**

1. Discover
2. Build
3. Iterate

各ステップ：
- アイコン
- 短い説明文（1〜2文）
- 実務っぽい具体語（仮説、検証、改善、制約など）

※ 「エンジニアでなくても、プロダクト思考がある」ことを示す

### 5-4. Artifacts / Process Proof セクション

- 「途中成果物」を見せるギャラリー
- 例：
  - 要件メモ
  - 構成図
  - テストログ
  - 改善履歴の断片

UI：
- 小さめカードをグリッド配置
- 完成物よりラフで良い
- キャプションは短く

### 5-5. About（超短縮）

- 3行程度の自己紹介
- 強みを箇条書き3点
- ツールやスキルは小さく補足程度

## 7. Case Study（詳細ページ）設計

**読み手が上から下にスクロールするだけで理解できる構造**

### 7-1. Hero
- プロジェクト名
- 一言成果
- 制作期間 / 役割
- 外部リンク（デモ等）

### 7-2. TL;DR セクション

カード3枚：
- Problem
- Solution
- Impact

※ 採用側がここだけ読んでも概要が掴めること

### 7-3. Context
- 背景
- 制約条件（時間・環境・前提）

### 7-4. Process

時系列セクション：
- Research
- Options（代替案と選定理由）
- Build
- Validate

※ コード説明は禁止、判断軸を書く

### 7-5. Screens
- UIキャプチャ 2〜4枚
- キャプション付き
- 実際に触れそうな雰囲気を重視

### 7-6. What I'd do next
- 改善案3つ
- 運用・拡張・リスク視点を含める

## 11. 重要な原則

1. **中身（文章・具体事例）は後から差し替える** - 今は構造・見た目・思想を正確に伝える
2. **技術スタック説明は一切しない** - UI・情報設計・プロジェクトの見せ方を最優先
3. **デザイン指示は実装者が誤解しない粒度で専門用語を使う**
4. **完成度よりも意図とスピードが伝わることを重視**
5. **コード説明は禁止、判断軸を書く**
